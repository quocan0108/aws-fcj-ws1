Thiết lập CI/CD cho project

Trong phần này, chúng ta sẽ thiết lập một CI/CD pipeline cơ bản cho một dự án Next.js sử dụng GitHub Actions. Mục tiêu là làm quen với cách tự động hóa quy trình từ khi commit mã nguồn đến khi triển khai ứng dụng lên môi trường sản xuất. Qua đó, bạn sẽ hiểu được cách GitHub Actions có thể giúp tối ưu hóa và tự động hóa các bước phát triển và triển khai trong dự án của mình.

Quy trình sẽ bao gồm các phần sau:

  1. Chuẩn bị môi trường làm việc bằng cách clone dự án mẫu Next.js và cấu hình các dependencies cần thiết.

  2. Tùy chỉnh và kiểm tra Unit Tests để đảm bảo mã nguồn được kiểm tra đầy đủ trước khi triển khai.

  3. Thiết lập GitHub Actions để tự động hóa quá trình build và kiểm thử mỗi khi có thay đổi mã nguồn.

  4. Triển khai ứng dụng lên GitHub Pages, giúp chúng ta dễ dàng kiểm tra và sử dụng ứng dụng sau khi được cập nhật.

Với những phần này, bạn sẽ nắm rõ cách triển khai một quy trình CI/CD đơn giản nhưng hiệu quả cho dự án của mình.

Bước 1: Clone dự án mẫu NextJS về máy

  • Link repository: https://github.com/vercel/next.js

Bước 2: Chọn folder examples/with-jest và mở trong terminal

  • Đây là một dự án Next.js đã được cấu hình sẵn với các unit test.

Bước 3: Cài đặt npm dependencies

  • Chạy lệnh npm install để cài đặt tất cả các dependencies cần thiết cho dự án.

Bước 4: Chạy project

  • Sử dụng lệnh npm run dev để khởi động server phát triển của Next.js.

Bước 5 Truy cập vào localhost:3000 để xem ứng dụng đang chạy.

Bước 6: Chỉnh sửa file package.json

  • Mở file package.json và thay đổi dòng “test”: “jest –watch” thành “test”: “jest –coverage” để xem tỷ lệ độ phủ của các unit test.

Bước 7: Chạy lệnh npm run test để thực hiện các bài kiểm thử.

Bước 8: Bảng báo cáo độ phủ của mã nguồn sẽ bao gồm các thông tin như:

  • File: Danh sách các tệp tin đã được kiểm tra.
  • % Stmts: Tỷ lệ phần trăm các câu lệnh (statements) đã được kiểm tra.
  • % Branch: Tỷ lệ phần trăm các nhánh điều kiện (branches) trong mã đã được kiểm tra.
  • % Funcs: Tỷ lệ phần trăm các hàm (functions) đã được kiểm tra.
  • % Lines: Tỷ lệ phần trăm các dòng mã đã được kiểm tra.
  • Uncovered Line #s: Các dòng mã chưa được kiểm tra.

Bước 9: Tuỳ chỉnh một số unit test

  • Mở file app/blog/page.tsx.
  • Thêm hoặc sửa đổi các unit test để bao quát thêm các chức năng hoặc kịch bản cần thiết.

Bước 10: Tiếp tục với file app/blog/page.test.tsx

  • Tiến hành sửa đổi trong file test app/blog/page.test.tsx để phù hợp với bất kỳ thay đổi nào đã thực hiện trong app/blog/page.tsx.

Bước 11: Chạy lại bộ kiểm thử

  • Chạy lệnh npm run test để kiểm tra các unit test.
  • Đảm bảo rằng tất cả các unit test đều thành công và có thêm ít nhất một unit test mới được bao phủ.

Bước 12: Đẩy project lên GitHub

  • Đảm bảo bạn đã commit tất cả các thay đổi trong Git và đẩy (push) lên repository GitHub của bạn.

Bước 13: Đẩy project lên GitHub thành công

  • Xác nhận rằng project đã được đẩy lên GitHub thành công bằng cách kiểm tra repository của bạn.

Bước 14: Trên repository, chọn Actions

  • Truy cập vào tab Actions trên GitHub repository của bạn để cấu hình GitHub Actions.

Bước 15: Tìm NodeJS và chọn Configure NodeJS

  • Tìm workflow NodeJS đã được cấu hình sẵn và chọn Configure NodeJS để chỉnh sửa hoặc thiết lập các bước tự động.

Bước 16: Giao diện Actions NodeJS

  • Tại giao diện này, bạn có thể thêm, sửa hoặc xóa các bước cần thiết cho quy trình CI/CD của bạn.

Bước 17: Chọn Commit Changes

  • Sau khi chỉnh sửa xong, chọn Commit Changes để lưu lại các thay đổi trong workflow của bạn.

Bước 18: Sau khi commit changes, trong repo sẽ có folder .github/workflows

  • Folder này chứa các file .yml để làm việc với GitHub Actions, trong đó bạn có thể xem và chỉnh sửa các công việc CI/CD.

Bước 19: Tạo và chuyển sang một branch khác: branch “update”

  • Tạo một branch mới tên là “update” và chuyển sang branch đó.

  • Thêm một vài dòng code hoặc thực hiện thay đổi cần thiết.

Bước 20: Đẩy code mới lên branch “update”

  • Đẩy (push) các thay đổi lên branch “update” trên GitHub.

Bước 21: Chọn Compare & Pull Request

  • Trên GitHub, chọn Compare & Pull Request để bắt đầu quá trình merge code từ branch “update” vào branch “main”.

Bước 22: Tạo pull request vào branch main

  • Điền thông tin cần thiết và tạo pull request để merge code vào branch chính (main).

Bước 23: Hệ thống tự động chạy những công việc đã liệt kê trong file nodejs.yml

  • Sau khi tạo pull request, GitHub Actions sẽ tự động chạy các công việc được cấu hình trong file nodejs.yml.

Bước 24: Vào Actions để kiểm tra chi tiết các công việc đã hoàn thành

  • Truy cập vào tab Actions để kiểm tra trạng thái và kết quả của các công việc CI/CD.

Bước 25: Vào Settings chọn Pages để Deploy

  • Vào mục Settings của repository, chọn Pages để cấu hình và triển khai ứng dụng của bạn.

Bước 26: Trong phần Source, chọn GitHub Actions

  • Trong phần cấu hình Source cho GitHub Pages, chọn GitHub Actions để sử dụng workflow đã thiết lập cho việc deploy.

Bước 27: Chọn Configure

  • Chọn Configure để hoàn tất cấu hình deploy thông qua GitHub Actions.

Bước 28: Chọn Commit Changes

  • Sau khi cấu hình xong, chọn Commit Changes để lưu lại các thay đổi.

Bước 29: Merge Pull Request

  • Sau khi các công việc kiểm thử và xây dựng trên GitHub Actions đã hoàn tất thành công, bạn cần merge pull request từ branch “update” vào branch “main”.

Bước 30: Merge

Bước 31: Tạo branch “production”

  • Chọn New Branch trong mục Code, tạo một branch mới có tên là “production”

Bước 32: Pull Request từ main đến production

  • Tạo một pull request để merge các thay đổi từ branch “main” sang branch “production”.
  • Truy cập vào repository của bạn trên GitHub, chọn Compare & Pull Request từ branch “main” đến branch “production”.

Bước 33: Thêm title “release” vào nội dung và chọn Create pull request

Bước 34: Thiết lập hệ thống CI/CD thành công

  • Sau khi bạn đã tạo và merge pull request từ “main” sang “production”, hệ thống CI/CD (GitHub Actions) sẽ tự động chạy các công việc để triển khai (deploy) ứng dụng từ branch “production”.
  • Kiểm tra các hoạt động này trong tab Actions để đảm bảo tất cả các bước đã được hoàn thành thành công và ứng dụng đã được triển khai lên môi trường production.
