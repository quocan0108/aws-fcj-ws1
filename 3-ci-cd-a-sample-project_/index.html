Setting up CI/CD for project

In this section, we will set up a basic CI/CD pipeline for a Next.js project using GitHub Actions. The goal is to get familiar with automating the process from committing code to deploying the application to a production environment. Through this, you will understand how GitHub Actions can help optimize and automate development and deployment steps in your project.

The process will include the following sections:

  1. Prepare the working environment by cloning a sample Next.js project and configuring the necessary dependencies.

  2. Customize and check Unit Tests to ensure the code is thoroughly tested before deployment.

  3. Set up GitHub Actions to automate the build and testing process whenever there is a code change.

  4. Deploy the application to GitHub Pages, allowing us to easily test and use the application after updates.

With these sections, you will clearly understand how to implement a simple yet effective CI/CD process for your project.

Step 1: Clone the sample NextJS project to your local machine

  • Repository link: https://github.com/vercel/next.js

Step 2: Select the folder examples/with-jest and open it in the terminal

  • This is a pre-configured Next.js project with unit tests.

Step 3: Install npm dependencies

  • Run the command npm install to install all necessary dependencies for the project.

Step 4: Run the project

  • Use the command npm run dev to start the Next.js development server.

Step 5: Access localhost:3000 to view the running application.

Step 6: Edit the package.json file

  • Open the package.json file and change the line “test”: “jest –watch” to “test”: “jest –coverage” to view the test coverage.

Step 7: Run the command npm run test to execute the tests.

Step 8: The source code coverage report will include information such as:

  • File: List of files that have been tested.
  • % Stmts: The percentage of statements that have been tested.
  • % Branch: The percentage of conditional branches in the code that have been tested.
  • % Funcs: The percentage of functions that have been tested.
  • % Lines: The percentage of code lines that have been tested.
  • Uncovered Line #s: The lines of code that have not been tested.

Step 9: Customize some unit tests

  • Open the file app/blog/page.tsx.
  • Add or modify unit tests to cover additional functionalities or scenarios.

Step 10: Continue with the file app/blog/page.test.tsx

  • Make changes in the test file app/blog/page.test.tsx to match any changes made in app/blog/page.tsx.

Step 11: Rerun the test suite

  • Run the command npm run test to check the unit tests.
  • Ensure that all unit tests pass and that there is at least one new unit test covered.

Step 12: Push the project to GitHub

  • Make sure you have committed all changes in Git and pushed them to your GitHub repository.

Step 13: Successfully pushed the project to GitHub

  • Confirm that the project has been successfully pushed to GitHub by checking your repository.

Step 14: On the repository, select Actions

  • Go to the Actions tab on your GitHub repository to configure GitHub Actions.

Step 15: Find NodeJS and select Configure NodeJS

  • Find the pre-configured NodeJS workflow and select Configure NodeJS to edit or set up automated steps.

Step 16: NodeJS Actions interface

  • In this interface, you can add, edit, or delete the necessary steps for your CI/CD process.

Step 17: Select Commit Changes

  • After editing, select Commit Changes to save your workflow changes.

Step 18: After committing changes, there will be a .github/workflows folder in the repository

  • This folder contains .yml files for working with GitHub Actions, where you can view and edit CI/CD tasks.

Step 19: Create and switch to a different branch: branch “update”

  • Create a new branch named “update” and switch to that branch.

  • Add a few lines of code or make necessary changes.

Step 20: Push new code to the “update” branch

  • Push the changes to the “update” branch on GitHub.

Step 21: Select Compare & Pull Request

  • On GitHub, select Compare & Pull Request to start merging code from the “update” branch into the “main” branch.

Step 22: Create a pull request to the main branch

  • Fill in the necessary information and create a pull request to merge the code into the main branch.

Step 23: The system automatically runs the jobs listed in the nodejs.yml file

  • After creating the pull request, GitHub Actions will automatically run the jobs configured in the nodejs.yml file.

Step 24: Go to Actions to check the details of the completed jobs

  • Access the Actions tab to check the status and results of the CI/CD jobs.

Step 25: Go to Settings, select Pages to Deploy

  • Go to the Settings section of the repository and select Pages to configure and deploy your application.

Step 26: In the Source section, select GitHub Actions

  • In the Source configuration for GitHub Pages, select GitHub Actions to use the configured workflow for deployment.

Step 27: Select Configure

  • Select Configure to finalize the deployment configuration through GitHub Actions.

Step 28: Select Commit Changes

  • After completing the configuration, select “Commit Changes” to save the changes.

Step 29: Merge Pull Request

  • After all the testing and build jobs in GitHub Actions have successfully completed, you need to merge the pull request from the “update” branch into the “main” branch.

Bước 30: Merge

Step 31: Create “production” branch

  • In the “Code” section, select “New Branch” to create a new branch named “production.”

Step 32: Pull Request from main to production

  • Create a pull request to merge changes from the “main” branch to the “production” branch.
  • Go to your repository on GitHub, select “Compare & Pull Request” from the “main” branch to the “production” branch.

Step 33: Add “release” title and select Create pull request

Step 34: Successfully set up CI/CD system

  • After creating and merging the pull request from “main” to “production,” the CI/CD system (GitHub Actions) will automatically run tasks to deploy the application from the “production” branch.

  • Check these activities in the Actions tab to ensure all steps have been completed successfully and the application has been deployed to the production environment.
